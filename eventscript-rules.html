What is EventScript?

EventScript is a declarative language for writing home automation rules. Each rule is an expression of the form condition => action. When the condition's triggers fire, the action runs.

-- When sunset occurs, turn on the outdoor lights
rule("@sunset => outdoorLights:on")

-- When temperature drops below 5°C, warn
rule("tempSensor:value < 5 => log('Frost warning!')")

Time Triggers @

rule("@08:00 => log('Good morning!')")         -- exact time
rule("@{07:00,19:00} => securityCheck()")      -- multiple times
rule("@sunset => outdoorLights:on")            -- sun event
rule("@sunrise-00:30 => blinds:open")          -- offset from sun
rule("@{08:00,catch} => morningLight:on")      -- catch missed fires
  • Use catch in the time list so the rule fires immediately on startup if the time has already passed today
  • Day guards: add & wday('mon-fri') to restrict to weekdays

Interval Triggers @@

rule("@@00:05 => log('Every 5 minutes')")
rule("@@00:00:30 => temperatureCheck()")   -- every 30 seconds
rule("@@-01:00 => log('Top of the hour')")  -- clock-aligned
  • Relative @@HH:MM — fires every N minutes from startup
  • Aligned @@-HH:MM — fires on the clock boundary (e.g. every :00)
  • Supports seconds: @@HH:MM:SS
  • Great for periodic polling, health checks, or scheduled reports

Device Triggers

rule("motionSensor:breached => hallLight:on")

  • :breached / :isOpen / :isOn / :value — common property triggers
  • Wrap multiple device IDs in {...} to trigger on any of them

Time Guards

-- Only activate night light between 22:00 and 06:00
rule("motion:breached & 22:00..06:00 => nightLight:on")

-- Only on weekdays
rule("@07:30 & wday('mon-fri') => log('Weekday!')")

-- Sustained condition: true for N minutes before firing
rule("trueFor(00:10, !hallMotion:breached) => hallLight:off")
  • Use & to add a guard expression to any trigger
  • HH:MM..HH:MM tests whether "now" is within that range
  • trueFor(duration, expr) waits until expr is stable for that long

Variables & the Home Table

local rule, var = er.rule, er.variables

var.threshold = 25
var.HT = { kitchen = { light = 54, motion = 77 } }

rule("@sunset => HT.kitchen.light:on")
rule("tempSensor:value > threshold => fan:on")

function var.securityCheck()
  if frontDoor:isOpen then log('Door open!') end
end
  • er.variables (alias var) — shared across all rules in scope
  • A Home Table organises device IDs into readable paths

Trigger Variables

local rule, triggerVar = er.rule, er.triggerVariables

triggerVar.homeOccupied = false

-- This rule fires whenever homeOccupied changes to true
rule("homeOccupied == true => log('Someone is home!')")

rule("@sunset => homeOccupied = true")
rule("motionSensor:breached => homeOccupied = true")
  • Writing to triggerVar.x posts a trigger-variable event, firing any watching rule
  • Unlike plain variables, these react — they are the subject of conditions

Async Actions: wait()

-- Turn light on, wait 5 min, turn off if no motion
rule([[motion:breached =>
  hallLight:on;
  wait(300000);           -- 300 000 ms = 5 minutes
  if !motion:breached then
    hallLight:off
  end
]])
  • wait(ms) suspends the action and resumes via setTimeout
  • Multiple wait calls chain correctly in sequence
  • Set rule.verbosity = "verbose" to log 💤 / ⏰ around each wait

Custom Events

-- Post a named event from a trigger
rule("@sunset => post(#eveningRoutine)")
rule("motion:breached & 22:00..06:00 => post(#nightMode)")

-- Handlers act like named subroutines
rule([[#eveningRoutine =>
  outdoorLights:on;
  securitySystem:arm;
  log('Evening routine activated')
]])
  • #name is a custom event — use post(#name) to fire it
  • Event rules decouple triggers from logic, keeping rules small and clear

Scheduling Events

-- Relative delay  (+/ prefix)
rule("@sunset => post(#lightsOff, +/01:30)")    -- 1h 30m later
rule("motion:breached => post(#autoOff, +/00:05)") -- 5min later

-- Absolute time  (t/ = today, n/ = today-or-tomorrow)
rule("@08:00 => post(#cleanup, n/10:00)")       -- next 10:00
rule("@sunset => post(#bedtime, t/23:00)")      -- 23:00 today
  • +/HH:MM — relative delay from now
  • t/HH:MM — absolute time today; n/HH:MM — today or tomorrow
  • Full dates: 2026/12/24/18:00 or /12/24/18:00 (this year)

Event Parameters

rule("temp:value > 25 => post(#alert{level='high'})")
rule("temp:value < 15 => post(#alert{level='low'})")

-- Pattern-match on parameters
rule("#alert{level='high'} => fan:on")
rule("#alert{level='low'}  => heater:on")
rule("#alert => log('Temp event: %s', level)") -- catch-all

rule("motion:breached => post(#motion{room='hall'})")
  • Parameters flow into the action scope as named locals
  • A rule without params is a catch-all — receives any matching event name

Cancel & Debounce

-- Each new motion event resets the 10-minute countdown
rule([[motion:breached =>
  cancel(lightTimer);                       -- safe if nil
  lightTimer = post(#autoOff, +/00:10);     -- reset timer
  hallLight:on
]])

rule([[#autoOff =>
  if !motion:breached then hallLight:off end
]])
  • cancel(ref) is safe to call on nil or an already-fired reference
  • This pattern keeps a light on as long as motion continues, then auto-off

Best Practices

  • Use a Home Table — organise device IDs so rules read like plain English
  • One rule, one responsibility — post events to break complex logic into subroutines
  • Add catch to morning time rules so they fire on restart if the time has passed
  • Align intervals with @@-HH:MM for on-the-clock reports
  • Use verbosity — set rule.verbosity = "verbose" while debugging, then revert
  • Cancel before post — always cancel(timer) before rescheduling to avoid duplicates
